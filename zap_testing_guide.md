# ZAP Testing Guide - DAST Learning Lab

> This guide explains what the Stage 5 workflow does and why each step exists.
> The goal is to make ZAP feel less like a magic button and more like a repeatable security test.

---

## 1. DAST vs SAST

**SAST** means Static Application Security Testing.
It reads source code without running the app.
It is like checking a recipe for unsafe ingredients before anyone cooks it.
For this project, a SAST tool would inspect `LoginController.java` and notice SQL built with string concatenation.

**DAST** means Dynamic Application Security Testing.
It tests the running application by sending real HTTP requests.
It is like tasting the cooked meal and checking what actually happens.
For this project, ZAP sends payloads to `http://localhost:8080/api/login`, `/api/greet`, `/api/echo`, and `/api/search`.

The Jenkins workflow uses DAST because the lesson is about scanning a live Spring Boot API the way an attacker would reach it over HTTP.

---

## 2. Manual ZAP Workflow

### Step 1: Start the backend

Run the Spring Boot app first because ZAP cannot scan source files.
DAST needs a reachable URL, so the backend must respond on `http://localhost:8080`.
In Jenkins, the `Start Backend` stage starts the JAR and waits for `/v3/api-docs`.

### Step 2: Import the OpenAPI spec

OpenAPI tells ZAP which endpoints and parameters exist.
The spec is generated by Springdoc at `http://localhost:8080/v3/api-docs`.
This is better than manually typing endpoint URLs because every new `@RestController` can appear in the spec automatically.

### Step 3: Spider the target

The spider crawls reachable URLs and adds them to ZAP's site tree.
In this API project, OpenAPI is the main discovery mechanism, while the spider is a useful extra discovery step.
The Stage 5 YAML keeps the spider in scope with the `DAST-API-Context`.

### Step 4: Wait for passive scanning

Passive scanning reads normal responses and looks for issues without changing requests.
Examples include missing headers, information disclosure, and permissive CORS.
The YAML uses `passiveScan-wait` so ZAP finishes reading responses before active attacks begin.

### Step 5: Run active scan

Active scanning changes requests by adding attack payloads.
This is where ZAP tests SQL injection and XSS.
Active scanning should only run against systems you own or have permission to test.

### Step 6: Generate the report

The report turns scan results into a readable HTML file.
The Stage 5 YAML writes `zap-reports/zap-automation-report.html`.
The Jenkinsfile publishes that file with the HTML Publisher plugin.

## 3. What ZAP Finds In This App

### SQL Injection - `LoginController.java`

`backend/src/main/java/com/dast/demo/controller/LoginController.java` builds SQL with user input:

```java
String sql = "SELECT * FROM users WHERE username='" + username
           + "' AND password='" + password + "'";
```

That is vulnerable because `username` and `password` are treated as SQL code.
ZAP rule `40018` sends payloads such as quotes, boolean conditions, and database-specific probes.
The H2 database returns real SQL behavior and errors, which helps ZAP confirm CWE-89.

### Reflected XSS - `XssController.java`

`backend/src/main/java/com/dast/demo/controller/XssController.java` returns `text/html` and inserts input into HTML.
The `/api/greet` endpoint places `name` inside a `<span>`.
The `/api/echo` endpoint places `input` inside a `<div>`.
ZAP rule `40012` sends script payloads and checks whether they return unescaped in HTML.

### Reflected Content In JSON - `SearchController.java`

`backend/src/main/java/com/dast/demo/controller/SearchController.java` reflects `query` inside a JSON response.
JSON is safer than HTML because browsers do not execute it as a page by default.
The reflection is still useful for teaching because it shows how context changes exploitability.

### Plaintext Passwords - `data.sql`

`backend/src/main/resources/data.sql` stores demo passwords in plain text.
ZAP will not usually find this from HTTP alone because it is a source/database design issue, not a visible runtime response.
This is a good interview example of why DAST and SAST complement each other.

## 4. How Jenkins Automates The Manual Steps

| Manual action | Jenkins stage | Why it exists |
|---|---|---|
| Copy project files | `Checkout` | Puts the project into the Jenkins workspace |
| Compile backend | `Build Backend` | Creates the executable Spring Boot JAR |
| Run unit tests | `Test Backend` | Publishes JUnit results before DAST |
| Start app | `Start Backend` | Gives ZAP a live target on port 8080 |
| Import OpenAPI | `ZAP Automation Scan` | YAML `openapi` job discovers API endpoints |
| Spider | `ZAP Automation Scan` | YAML `spider` job crawls in-scope URLs |
| Passive scan | `ZAP Automation Scan` | YAML `passiveScan-wait` job waits for analysis |
| Active scan | `ZAP Automation Scan` | YAML `activeScan` job runs SQLi/XSS rules |
| Export report | `ZAP Automation Scan` | YAML `report` job writes HTML |
| View report | `Publish ZAP Report` | Jenkins displays the HTML report |
| Stop app | `Stop Backend` and `post always` | Frees port 8080 for the next run |

The important workflow change is that Jenkins no longer hardcodes endpoint scanning with `-quickurl`.
Jenkins now runs the ZAP plan with `-autorun`, and the ZAP plan imports the API from `/v3/api-docs`.
That makes the pipeline scale as the API grows.

## 5. How To Read ZAP Alerts

### High

High alerts usually mean the issue is directly exploitable.
In this lab, SQL Injection on `/api/login` should be treated as High because it can bypass authentication.
Start remediation with High alerts first.

### Medium

Medium alerts are important but may need context to judge exploitability.
Reflected XSS often appears as Medium or High depending on confidence and browser execution context.
Missing security headers may also appear here.

### Low

Low alerts are weaker risks or hard-to-exploit issues.
They should still be reviewed because several Low issues can combine into a bigger weakness.
Examples include minor information leaks or weak browser-hardening headers.

### Informational

Informational alerts teach you something about the app but are not usually vulnerabilities by themselves.
Examples include technology fingerprints or interesting responses.
Use them to understand the attack surface.

## 6. Why Rule IDs 40018, 40012, And 40014 Matter

`40018` is ZAP's SQL Injection rule.
It maps directly to the vulnerable SQL string in `LoginController.java`.

`40012` is ZAP's reflected XSS rule.
It maps directly to the raw HTML responses in `XssController.java`.

`40014` is ZAP's persistent XSS rule.
This lab does not store user-submitted HTML permanently, but enabling it shows how the same scan policy would cover stored XSS when future endpoints are added.

The YAML keeps the default policy at medium strength and explicitly raises the SQLi and reflected XSS rules to high strength so the lab's intended findings are easier to reproduce.
